`v0.5` should treat `cx bundle --include-doc-exports` as the final bundle-integration goal for Antora-derived review exports, with `.mmd.txt` bundle outputs, and it should remain opt-in only.

## What

Add an explicit bundle flag such as `--include-doc-exports` so operators can
request that generated docs review exports be written into the bundle
folder.

This is the bridge between a dedicated docs export workflow and later deeper
bundle integration.

## Why

An opt-in flag lets the project prove that bundle-adjacent doc exports are
useful before changing the default semantics of `cx bundle`.

That keeps the proof-path bundle contract conservative during `v0.5` while
still letting interested operators exercise the combined workflow and produce
LLM-reviewable final docs beside the ordinary bundle outputs.

## How

For `v0.5`, the bundle-facing rules should be:

- `cx docs export` remains the primary docs export entrypoint
- `cx bundle --include-doc-exports` is optional, not implied
- exported `.mmd.txt` bundle files are added only when requested
- trust language remains explicit about derived review exports
- any manifest recording should avoid pretending the exports already have full
  proof-artifact semantics

This keeps the default `cx bundle` path stable while allowing real operator
feedback on the integrated mode.
