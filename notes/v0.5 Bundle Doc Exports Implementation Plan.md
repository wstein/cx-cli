`v0.5` should implement `cx bundle --include-doc-exports` as a staged opt-in bridge that reuses a dedicated docs export path, records explicit derived-export metadata, and extends bundle validation without prematurely defaulting the behavior on.

## What

Implement the `v0.5` goal in six bounded phases:

- phase 1: add a reusable docs export engine
- phase 2: add a user-facing `cx docs export` command
- phase 3: add `cx bundle --include-doc-exports`
- phase 4: record derived doc exports in bundle metadata
- phase 5: extend validation, update-mode sync, and checksums
- phase 6: harden with tests, docs, and operator-facing JSON output

## Why

The bridge only works well if docs export is a first-class implementation unit
before bundle integration depends on it.

That keeps the design testable and lets `v0.5` prove operator value without
forcing the full default-on bundle semantics that belong to the later `v0.6`
decision.

## How

Phase 1: reusable docs export engine

- extract Antora export behavior out of `scripts/build-antora-site.js`
- create a library module under `src/docs/` or similar for:
  - export specs
  - output naming
  - export execution
  - provenance metadata collection
- define the `.mmd.md` export naming contract
- expand export coverage beyond the current html-single examples where needed

Likely touch points:

- `scripts/build-antora-site.js`
- new `src/docs/*`
- tests for deterministic export inventory and naming

Phase 2: `cx docs export`

- add a `docs export` CLI command in `src/cli/main.ts`
- implement command logic in `src/cli/commands/docs.ts` or equivalent
- support explicit output directory selection
- support human output and `--json`
- return per-export metadata:
  - logical surface name
  - output path
  - hash
  - size
  - source module or nav root

Acceptance criteria:

- operators can run docs export without bundling
- output is reviewable and deterministic for a clean tree

Phase 3: opt-in bundle bridge

- add `--include-doc-exports` to `cx bundle`
- keep the flag default `false`
- run docs export only after core bundle rendering succeeds
- stage exported files into the active bundle directory
- choose a stable subdirectory such as `{project}-docs-exports/`

Why this order:

- preserves the default bundle path
- avoids making bundle correctness depend on docs export unless requested

Phase 4: manifest metadata

- add a dedicated manifest field for derived doc exports
- do not overload `sections` or `assets`
- include for each export:
  - surface name
  - stored path
  - sha256
  - size
  - generator identity
  - source provenance
  - trust classification as derived review export

Likely touch points:

- `src/manifest/types.ts`
- `src/manifest/build.ts`
- `src/manifest/json.ts`
- `schemas/manifest-*.schema.json`

Phase 5: validate, checksums, update sync

- include doc-export files in the checksum sidecar when present
- teach `validateBundle(...)` that recorded doc exports must exist
- include them in `expectedFiles` for `--update` prune and sync behavior
- keep verification semantics conservative in `v0.5`
- verify presence and checksum integrity, but do not claim source-tree proof
  parity for docs exports unless the contract is explicitly defined

Likely touch points:

- `src/cli/commands/bundle.ts`
- `src/bundle/validate.ts`
- `src/manifest/checksums.ts`
- `src/bundle/verify.ts`

Phase 6: tests, docs, operator outputs

- unit tests for export metadata and naming
- CLI tests for `cx docs export`
- bundle tests for `--include-doc-exports`
- contract tests for manifest schema and checksum inclusion
- update-mode tests for pruning removed exports
- manual/config docs for the new command and flag
- release notes and examples

Suggested acceptance tests:

- `cx docs export` emits `.mmd.md` files with stable names
- `cx bundle` without the flag emits no doc exports
- `cx bundle --include-doc-exports` writes the export files into the bundle
- manifest metadata appears only when the flag is used
- checksum file includes the recorded exports
- `cx validate` fails when a recorded export file is missing
- `cx bundle --update --include-doc-exports` prunes obsolete exports
