`v0.5` should introduce a dedicated docs export workflow for Antora-derived MultiMarkdown outputs instead of hiding that behavior inside default bundling.

## What

The preferred operator surface is a command such as `cx docs export` that
builds per-surface `.mmd.md` outputs for architecture, manual, onboarding,
workflow, release, and similar canonical docs classes.

This command should make the export intent explicit and keep the Antora build
logic readable, testable, and separately evolvable from native proof-path
bundling.

## Why

The export is valuable because downstream reviewers and LLMs benefit from the
final rendered documentation surface, not just the source `.adoc` tree.

A dedicated workflow keeps that benefit without forcing every `cx bundle`
execution to own the full Antora export dependency chain, failure mode, and
trust implication on day one.

## How

For `v0.5`, prefer this sequence:

- add `cx docs export`
- emit `.mmd.md` files per canonical docs surface
- write them to a stable reviewable folder
- optionally allow `cx bundle --include-doc-exports` later as an opt-in bridge
- if manifest references are added, model them as derived review exports first

This keeps room for a later contract promotion without pretending that the
promotion already happened.
