`v0.6` can make doc-export integration the default `cx bundle` behavior, so `.mmd.txt` exports become first-class bundle content rather than an opt-in add-on.

## What

Promote the `v0.5` opt-in bridge into the default behavior:

- `cx bundle` generates Antora-derived `.mmd.txt` docs exports automatically
- the exported docs live inside the bundle folder
- the docs exports are treated as first-class bundle content

This is the point where the bundle contract would stop treating them as a mere
optional sidecar workflow.

## Why

If the `v0.5` opt-in phase proves stable, deterministic, and useful, default
bundle integration simplifies the operator story and makes final rendered docs
part of the normal `cx` handoff experience.

That would align the bundle more closely with the broader `cx` goal of making
reviewable downstream artifacts available to both humans and LLMs without extra
workflow branching.

## How

Do not ship this by default until the `v0.5` bridge has already proven:

- stable export generation
- acceptable bundle runtime cost
- clear provenance metadata
- manifest semantics for first-class docs content
- checksum, validate, verify, and prune behavior for the new files

When those conditions are met, move the default on in `v0.6` and update docs,
templates, tests, and trust language together so the promoted semantics are
real rather than implied.
