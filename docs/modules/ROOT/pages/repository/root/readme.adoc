<p>cx-cli is a repository-native toolchain for agent-assisted development: it exposes live MCP workspace access, maintains durable project notes, and produces verifiable bundle artifacts for AI handoff. Its goal is to keep exploration, memory, and proof aligned so agents, developers, and CI can inspect the same trusted project context over time.</p>
<p>Mode chooser: <a href="https://wstein.github.io/cx-cli/docs/cx/0.5/manual/operating-modes/">Operating Modes</a></p>
<h2>Start Here</h2>
<p>Run <code>cx mcp</code>.</p>
<p>That is the shortest path to value:</p>
<ol>
<li>Run <code>cx mcp</code></li>
<li>Watch the agent work on live code</li>
<li>Learn the model later in <a href="https://wstein.github.io/cx-cli/docs/cx/0.5/manual/operating-modes/">Operating Modes</a> and <a href="https://wstein.github.io/cx-cli/docs/cx/0.5/architecture/mental-model/">Mental Model</a></li>
</ol>
<p>If you only remember one mental model, keep this one:</p>
<ul>
<li><code>cx mcp</code> is the live workspace surface.</li>
<li><code>cx notes</code> is the durable cognition layer.</li>
<li><code>cx bundle</code> and <code>cx verify</code> turn live work into a trusted artifact handoff.</li>
</ul>
<p>For the one-page map from hypothesis to memory to snapshot to proof, start with <a href="https://wstein.github.io/cx-cli/docs/cx/0.5/architecture/system-map/">System Map</a>.</p>
<h2>Choose The Surface</h2>
<h3>Live Agent Exploration</h3>
<p>Use this path when an LLM agent needs live workspace access, note maintenance, or targeted search.</p>
<ul>
<li>Key commands: <code>cx mcp</code>, <code>cx notes</code>, <code>cx doctor mcp</code>, <code>cx audit summary</code></li>
<li>Goal: expose the live workspace and note graph through MCP</li>
<li>Outcome: faster investigation without freezing a bundle too early</li>
</ul>
<h3>Deterministic Artifact Operations</h3>
<p>Use this path when the work needs deterministic bundles, artifact integrity, and hard failure semantics.</p>
<ul>
<li>Key commands: <code>cx bundle</code>, <code>cx verify</code>, <code>cx extract</code>, <code>cx validate</code></li>
<li>Goal: produce immutable, bit-for-bit verifiable bundle artifacts with SHA-256 sidecars</li>
<li>Outcome: a locked manifest that later humans, CI, and automation can trust</li>
</ul>
<h2>Why The Split Exists</h2>
<p><code>cx</code> solves the harder follow-through problem. It is for repositories where interactive agent help, durable notes, and reproducible handoffs all need to agree about what was trusted, what was observed, and what was proven.</p>
<p>That changes the design:</p>
<ul>
<li>the native kernel owns proof-path rendering, manifests, spans, and verification</li>
<li><code>cx mcp</code> helps agents inspect live code now.</li>
<li><code>cx notes</code> preserves reasoning that should survive the session.</li>
<li><code>cx bundle</code> and <code>cx verify</code> freeze and prove the exact handoff later.</li>
<li>the adapter/oracle seam remains only for diagnostics and parity visibility, not the shipped proof path</li>
</ul>
<p>The strictness is the feature. If a file lands in two sections, if a checksum drifts, or if extraction becomes approximate instead of deterministic, <code>cx</code> makes that visible instead of quietly proceeding.</p>
<h2>What You Get</h2>
<ul>
<li>VCS-driven master file list with git, fossil, hg, or filesystem fallback</li>
<li>Section planning that sorts tracked files without inventing new inputs</li>
<li>Catch-all sections for unmatched tracked files</li>
<li>Dirty-state guardrails that block tracked-file drift unless you opt in with <code>--force</code></li>
<li>Differential <code>--update</code> bundling that prunes orphaned outputs safely</li>
<li>One rendered output per section plus a shared handoff index</li>
<li>Persistent token accounting, note summaries, and behavioral lock data in the manifest</li>
<li>SHA-256 checksums and manifest-aware <code>inspect</code>, <code>list</code>, <code>validate</code>, <code>verify</code>, and <code>extract</code></li>
<li>Guided overlap diagnosis and repair with <code>cx doctor</code></li>
<li><code>cx notes</code> commands for note lifecycle, graph inspection, backlinks, and orphans</li>
<li>Structured <code>--json</code> output for CI and automation</li>
</ul>
<h2>Operator Surfaces</h2>
<p><code>cx</code> now exposes four distinct operator surfaces:</p>
<ul>
<li>native proof path: <code>cx bundle</code>, <code>cx validate</code>, <code>cx verify</code>, <code>cx extract</code></li>
<li>live workspace path: <code>cx mcp</code></li>
<li>durable cognition path: <code>cx notes</code></li>
<li>adapter/oracle path: diagnostics and parity only, not the primary operator workflow</li>
</ul>
<h2>Quick Start</h2>
<ol>
<li><code>cx init --name demo</code></li>
<li><code>cx inspect --token-breakdown</code></li>
<li><code>cx mcp</code> for the live workspace path</li>
<li><code>cx bundle --config cx.toml</code> for the immutable artifact path</li>
</ol>
<p>Start with the same timeline in mind: investigate live, preserve durable reasoning, then freeze the handoff once it needs proof.</p>
<p>The invariants that protect downstream automation are explicit:</p>
<ul>
<li>SHA-256 checksums prove the emitted artifacts were not silently edited.</li>
<li>The manifest preserves the exact file inventory, note summaries, token budget, and provenance that later jobs need.</li>
<li><code>verify</code> compares the bundle back to a source tree so drift becomes explicit evidence.</li>
<li>Dirty-state gating stops tracked-file changes from masquerading as reviewed inputs.</li>
</ul>
<p>Trust shorthand:</p>
<ul>
<li>Source tree: trusted</li>
<li>Notes: conditional</li>
<li>Agent output: untrusted until verified</li>
<li>Bundle: trusted</li>
</ul>

<h2>Quick Start</h2>
<p>Initialize a starter config:</p>
<pre><code class="language-bash">cx init --name demo
</code></pre>
<p><code>cx init</code> writes <code>cx.toml</code>, <code>cx-mcp.toml</code>, <code>.editorconfig</code>, a workspace-aware <code>Makefile</code>, and a <code>notes/</code> directory with:</p>
<ul>
<li><code>Makefile</code> as a workspace-level entry point for native builds, testing, project cleanup, and workspace tasks; the template switches to language-specific recipes for Go, Rust, TypeScript/Node.js, Python, Java, Elixir, Julia, and Crystal when those markers are present</li>
<li><code>cx-mcp.toml</code> as the default MCP overlay for agent workflows; it extends <code>cx.toml</code> and keeps the MCP workspace config colocated with the repo root</li>
<li><code>.editorconfig</code> as the shared editor baseline for whitespace and line-ending defaults</li>
<li><code>notes/README.md</code> as the repository notes guide</li>
<li><code>notes/Templates/Atomic Note Template.md</code> as the atomic note template</li>
</ul>
<p>By default, <code>cx init</code> preserves existing init files and creates only the missing targets. Use <code>--force</code> to overwrite existing generated files.</p>
<p>Supported init templates include: <code>base</code>, <code>rust</code>, <code>go</code>, <code>typescript</code>, <code>python</code>, <code>java</code>, <code>elixir</code>, <code>julia</code>, and <code>crystal</code>.</p>
<p>You can explicitly choose an init template name with:</p>
<pre><code class="language-bash">cx init --name demo --template typescript
</code></pre>
<p>To list supported templates:</p>
<pre><code class="language-bash">cx init --template-list
</code></pre>
<p>When <code>--template</code> is omitted, <code>cx init</code> autodetects the workspace environment from files like <code>package.json</code>, <code>go.mod</code>, <code>pyproject.toml</code>, <code>pom.xml</code>, and <code>Cargo.toml</code>.</p>
<p>The generated notes directory is intentionally part of the repository contract. Keep architectural intent, implementation decisions, and durable project memory close to the code that depends on them.</p>
<h3>Development workflow</h3>
<ul>
<li><code>make test</code> or <code>bun run test</code> runs the fast Vitest unit suite.</li>
<li><code>make verify</code> or <code>bun run verify</code> runs lint, typecheck, build, the Vitest coverage lane, and the Bun compatibility smoke.</li>
<li><code>bun run ci:test:coverage</code> measures the full shared suite through Vitest and publishes the repository coverage artifacts.</li>
<li><code>bun run test:contracts</code> runs the contract suite through Vitest.</li>
<li><code>bun run test:all</code> runs the full shared suite through Vitest without collecting coverage.</li>
<li><code>bun run test:all:full</code> runs the authoritative Vitest coverage lane.</li>
<li><code>bun run test:vitest:bundle</code> isolates the slow bundle workflow, extraction, render-adapter, and JSON CLI suites.</li>
<li><code>bun run test:vitest:notes</code> isolates the slow notes graph and command suites, plus linked-note planning and operator contracts.</li>
<li><code>bun run test:vitest:planning</code> isolates planning and schema-integration suites for overlap, linked-note, and config-shape work.</li>
<li><code>bun run test:vitest:mcp</code> runs the focused MCP-heavy Vitest lane for server, policy, audit, and CLI MCP debugging.</li>
<li><code>bun run test:vitest:mcp:ui</code> opens the MCP lane in Vitest UI so you can rerun failures interactively and inspect coverage or import cost inside the MCP stack.</li>
<li><code>bun run test:vitest:mcp:adversarial</code> isolates startup hangs, malformed runtime payloads, and other hostile MCP failure modes in a smaller cockpit.</li>
<li>The <code>Makefile</code> delegates to <code>package.json</code> scripts using 1:1 shell shim wrappers.</li>
<li><code>package.json</code> does not expose an explicit <code>coverage</code> script; coverage is collected by test executions.</li>
</ul>
<h3>Editor autocomplete and linting</h3>
<p>The generated <code>cx.toml</code> includes a schema directive for <a href="https://taplo.tamasfe.dev/">Taplo</a> (the TOML extension in VS Code). This enables real-time autocomplete, validation, and linting:</p>
<pre><code class="language-toml">#:schema https://wstein.github.io/cx-cli/schemas/cx-config-v1.schema.json
schema_version = 1
project_name = &quot;myproject&quot;
</code></pre>
<p>The schema validates structural shape and enum constraints. Runtime validation in <code>cx load</code> enforces relational invariants (e.g., catch-all restrictions). See <a href="https://wstein.github.io/cx-cli/docs/cx/0.5/manual/config-reference/#json-schema-for-editor-tooling">Configuration Reference</a> for details.</p>
<p>For external tooling, use the published Pages endpoints:</p>
<ul>
<li><code>https://wstein.github.io/cx-cli/schemas/cx-config-v1.schema.json</code></li>
<li><code>https://wstein.github.io/cx-cli/schemas/cx-config-overlay-v1.schema.json</code></li>
</ul>
<p>The npm package also ships <code>schemas/</code> so offline consumers can keep using the local files without depending on the public site.</p>
<p>Preview the deterministic plan before writing anything:</p>
<pre><code class="language-bash">cx inspect --config cx.toml
</code></pre>
<p>If you want to see which section is carrying the most token budget, use:</p>
<pre><code class="language-bash">cx inspect --config cx.toml --token-breakdown
</code></pre>
<p>Build the bundle:</p>
<pre><code class="language-bash">cx bundle --config cx.toml
</code></pre>
<p>Apply a differential update (copy changed files only, prune orphaned artifacts):</p>
<pre><code class="language-bash">cx bundle --config cx.toml --update
</code></pre>
<p><code>--update</code> stages artifacts in a temporary OS directory first, then syncs the
result into <code>output_dir</code> so the final bundle only sees a deterministic, already
validated tree. Pruning is safety-gated: <code>cx</code> refuses to prune if the target
directory does not look like an existing bundle directory.</p>
<p>The differential algorithm is intentionally conservative. It is designed to
remove bundle artifacts that disappeared from the manifest while preserving the
rest of the output directory untouched.</p>
<p>Validate and verify it:</p>
<p>Assuming your config writes to <code>dist/demo-bundle</code>:</p>
<pre><code class="language-bash">cx validate dist/demo-bundle
cx verify dist/demo-bundle --against .
</code></pre>
<p>List stored files:</p>
<pre><code class="language-bash">cx list dist/demo-bundle
</code></pre>
<p>Extract one file back out:</p>
<pre><code class="language-bash">cx extract dist/demo-bundle --file src/index.ts --to /tmp/restore
</code></pre>
<h2>Typical CI Flow</h2>
<pre><code class="language-bash">CX_STRICT=true cx bundle --config cx.toml
cx verify dist/myproject-bundle --against . --config cx.toml
</code></pre>
<p><code>CX_STRICT=true</code> forces all configurable Category B behaviors to fail fast. That is the safest default for automated pipelines.</p>
<p>If an operator ever uses <code>--force</code> outside CI, add a manifest quarantine step
that rejects <code>forced_dirty</code> bundles before promotion. The operator manual
includes ready-to-use <code>jq</code> and Node.js examples.</p>

<h2>Active Ecosystem</h2>
<p><code>cx</code> is designed to participate in agentic workflows, not just produce static archives.</p>
<ul>
<li>The manifest records enough metadata for downstream tooling to route by note ID and section without reparsing Markdown.</li>
<li>The notes graph commands (<code>cx notes backlinks</code>, <code>cx notes orphans</code>, <code>cx notes code-links</code>) make the repository&#39;s knowledge layer queryable from the CLI.</li>
<li><code>cx mcp</code> starts the CX MCP server using <code>cx-mcp.toml</code> when available and falls back to <code>cx.toml</code> for the baseline agent profile. The server exposes native file-based <code>list</code>, <code>grep</code>, and <code>read</code> tools, live planning helpers (<code>inspect</code>, <code>bundle</code>, <code>doctor_mcp</code>, <code>doctor_workflow</code>), workspace-bound bundle recovery through <code>extract</code>, and note-native tools for reading, searching, creating, updating, renaming, deleting, and graph-inspecting repository notes over the active workspace scope.</li>
<li><code>cx doctor mcp</code> and <code>cx doctor secrets</code> provide deterministic diagnostics for the MCP inheritance boundary and the master-list secret scan.</li>
</ul>
<p>That means an LLM agent can ask for more context, retrieve just the relevant context surface, and reason about the repo as a live system rather than a static artifact.</p>
<h2>The Important Failure Model</h2>
<p>Some constraints are non-negotiable:</p>
<ul>
<li>section overlap is a hard failure when <code>dedup.mode = &quot;fail&quot;</code>; this is the default</li>
<li>asset collisions are hard failures</li>
<li>missing core adapter contract is a hard failure</li>
<li>degraded extraction is blocked unless you explicitly pass <code>--allow-degraded</code></li>
</ul>
<p>This is intentional. <code>cx</code> is designed to stop a pipeline before a bad bundle turns into a harder-to-debug downstream failure. For packed text files, the bundle hash tracks the normalized rendered output, so <code>verify</code> and <code>extract</code> stay aligned with the actual handover payload instead of source-byte exactness.</p>

<h2>LLM-Friendly Output Extensions</h2>
<p>By default, section outputs use these extensions:</p>
<ul>
<li>XML: <code>.xml.txt</code></li>
<li>JSON: <code>.json.txt</code></li>
<li>Markdown: <code>.md</code></li>
<li>Plain text: <code>.txt</code></li>
</ul>
<p>This keeps uploads frictionless across stricter web-based LLM file handlers.
You can override any of them in <code>cx.toml</code>:</p>
<pre><code class="language-toml">[output.extensions]

<h2>Development</h2>
<p>The repository-local <code>make</code> commands are intentionally small and explicit:</p>
<table>
<thead>
<tr>
<th>Command</th>
<th>Use it when</th>
</tr>
</thead>
<tbody><tr>
<td><code>make test</code></td>
<td>You want the fast unit loop while iterating</td>
</tr>
<tr>
<td><code>make verify</code></td>
<td>You want the normal local gate before merging</td>
</tr>
<tr>
<td><code>make release VERSION=vX.Y.Z</code></td>
<td>You are stepping through the two-phase release wizard</td>
</tr>
</tbody></table>
<pre><code class="language-bash">bun install
make test
make verify
make release VERSION=v0.4.0
make release VERSION=v0.4.0
</code></pre>
<p><code>make test</code> runs the fast unit suite. <code>make verify</code> is the normal local gate:
lint, typecheck, build, the Vitest coverage lane, and Bun compatibility smoke.
<code>bun run test:all</code> and <code>bun run test:all:full</code> stay on the native Vitest suite
and coverage lanes, while Bun remains in CI only for explicit compatibility
smoke. <code>make release</code>
now acts as a two-phase wizard: the first call with a new semantic version starts the
release candidate on <code>develop</code>, and the second call with that same version creates and pushes
the tag after CI is green.</p>
<p>If you prefer the underlying Bun commands directly:</p>
<pre><code class="language-bash">bun run ci:test:coverage
bun run test
bun run verify
VERSION=v0.4.0 bun run release
VERSION=v0.4.0 bun run release
</code></pre>
