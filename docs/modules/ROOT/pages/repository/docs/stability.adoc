= Stability Tiers
:page-layout: default


++++
<p>cx-cli tools are classified into four stability tiers, each with distinct guarantees and expected usage patterns. The tier of each tool appears in its MCP description with the format <code>[TIER_NAME]</code> (e.g., <code>[STABLE]</code>). For machine-readable inspection, prefer <code>cx mcp catalog --json</code>; <code>cx doctor mcp --json</code> also includes the same registered tool catalog when you need broader MCP diagnostics.</p>
<h2>Tier Definitions</h2>
<h2>Current Reliability Focus</h2>
<p>The current stability push is intentionally scoped to global-state reduction,
not broad toolchain replacement. <code>cx</code> continues to standardize on <code>yargs</code>,
<code>bun:test</code>, Bun-first CI execution, and the Repomix fork while reliability work
targets command I/O boundaries, workspace context injection, and clearer test
separation between pure behavior and integration seams.
CI now declares a minimum supported Bun runtime (<code>1.3.11</code>) and continuously
tests both the minimum and <code>latest</code> Bun lanes to detect regressions across the
supported runtime envelope.
The fast CI lane is intentionally unit-only and guarded by
<code>bun run ci:guard:fast-lane</code> so integration suites do not silently leak into
the low-latency feedback path.
Fast-lane runtime is also monitored with warning-first regression policy:
<code>bun run ci:test:fast:monitored</code> records timing drift and fails only on
sustained significant regressions.
The workflow graph is gated behind <code>test-fast</code>: downstream CI lanes do not
start if the fast gate is red, and CI artifacts are uploaded only after the
full lane set passes.</p>
<h3>STABLE</h3>
<p><strong>Guarantee:</strong> Tools in this tier have a documented, semver-protected API contract. Breaking changes require a major version bump of cx-cli.</p>
<p><strong>Output Schema:</strong> The JSON output structure is locked. Field names, types, and presence are guaranteed to remain consistent across minor and patch releases.</p>
<p><strong>Deprecation:</strong> Deprecated features will be signaled 2+ releases in advance with clear migration guidance.</p>
<p><strong>Examples:</strong> <code>list</code>, <code>grep</code>, <code>read</code>, <code>inspect</code>, <code>bundle</code>, <code>notes_new</code>, <code>notes_read</code>, <code>notes_update</code>, <code>notes_delete</code>, <code>notes_rename</code>, <code>notes_search</code>, <code>notes_list</code>, <code>notes_backlinks</code>, <code>notes_orphans</code>, <code>notes_code_links</code>, <code>notes_links</code>, <code>notes_graph</code></p>
<p><strong>Use in:</strong> Production CI/CD pipelines, long-lived scripts, documented integrations.</p>
<h3>BETA</h3>
<p><strong>Guarantee:</strong> Tools in this tier are functional and actively used, but their schema and behavior may evolve in minor releases (e.g., output fields added, removed, or renamed).</p>
<p><strong>Output Schema:</strong> The core functionality is stable, but the JSON structure may change between minor versions. Operators should write defensive parsers or pin to a specific version.</p>
<p><strong>Deprecation:</strong> Planned breaking changes will be announced 1 release in advance.</p>
<p><strong>Examples:</strong> <code>doctor_mcp</code>, <code>doctor_workflow</code>, <code>doctor_overlaps</code>, <code>doctor_secrets</code>, <code>replace_repomix_span</code></p>
<p><strong>Use in:</strong> Scripts that can tolerate schema changes, exploratory automation, diagnostic/observability workflows.</p>
<h3>EXPERIMENTAL</h3>
<p><strong>Guarantee:</strong> Tools in this tier are under active development. APIs and behavior are subject to change or removal without notice.</p>
<p><strong>Deprecation:</strong> No advance notice is provided; experimentals may be removed in any release.</p>
<p><strong>Use in:</strong> Testing, proof-of-concept code, internal tooling only.</p>
<h3>INTERNAL</h3>
<p><strong>Guarantee:</strong> Tools in this tier are not advertised publicly via MCP. They are used internally by cx-cli tests or development workflows.</p>
<p><strong>Use in:</strong> Tests and internal development only; external tooling should not depend on these.</p>

<h2>Versioning Policy</h2>
<table>
<thead>
<tr>
<th>Scenario</th>
<th>Tier</th>
<th>Action</th>
</tr>
</thead>
<tbody><tr>
<td>Add a new field to a STABLE tool&#39;s output</td>
<td>STABLE</td>
<td>Requires major version bump</td>
</tr>
<tr>
<td>Rename a field in a BETA tool&#39;s output</td>
<td>BETA</td>
<td>Allowed in minor version bump</td>
</tr>
<tr>
<td>Remove a BETA tool entirely</td>
<td>BETA</td>
<td>Requires minor version bump with 1-release deprecation notice</td>
</tr>
<tr>
<td>Remove a STABLE tool</td>
<td>STABLE</td>
<td>Requires major version bump with 2+ release deprecation notice</td>
</tr>
<tr>
<td>Add a new tool</td>
<td>N/A</td>
<td>Launched at current tier; increment minor version</td>
</tr>
</tbody></table>
<hr>
<h2>Migration Guide</h2>
<p>If a tool transitions from BETA to STABLE or is deprecated, the release notes will include:</p>
<ol>
<li>The change (e.g., &quot;doctor_workflow output schema is now stable&quot;)</li>
<li>Any schema changes in that version</li>
<li>Guidance for operators updating automation</li>
</ol>
