= MCP Tool Intent Taxonomy
:page-layout: default


++++
<p>This document is machine-oriented. It is intended for prompt engineering and agent policy guidance, not for primary human documentation.</p>
<p>See: <a href="https://github.com/wstein/cx-cli/blob/main/docs/modules/ROOT/pages/architecture/mental-model.adoc">MENTAL_MODEL.md</a> for the canonical CX triad, Track A vs Track B, and policy-tier model that this taxonomy sits underneath.</p>
<p><code>cx mcp</code> exposes tools in three intent-focused groups:</p>
<ul>
<li><code>Read / Observe</code> tools: safe inspection, search, and non-mutating note reads.</li>
<li><code>Plan / Preview</code> tools: deterministic planning and bundle metadata previews.</li>
<li><code>Write / Mutate</code> tools: state changes, edits, and new note or bundle actions.</li>
</ul>
<p>Use this taxonomy when building system prompts, tool selectors, or agent policies to enforce deterministic behavior and reduce accidental state changes.</p>
<h2>Read / Observe Tools</h2>
<p>These tools should be exposed to agents when the task is exploratory, investigative, or review-only. They do not change workspace files, notes, or bundle state.</p>
<ul>
<li><code>list</code></li>
<li><code>grep</code></li>
<li><code>read</code></li>
<li><code>doctor_mcp</code></li>
<li><code>doctor_workflow</code></li>
<li><code>doctor_overlaps</code></li>
<li><code>doctor_secrets</code></li>
<li><code>notes_read</code></li>
<li><code>notes_search</code></li>
<li><code>notes_list</code></li>
<li><code>notes_backlinks</code></li>
<li><code>notes_orphans</code></li>
<li><code>notes_code_links</code></li>
<li><code>notes_links</code></li>
</ul>
<h2>Plan / Preview</h2>
<p>These tools produce deterministic plans or bundle metadata without mutating the workspace.</p>
<ul>
<li><code>inspect</code></li>
<li><code>bundle</code></li>
</ul>
<h3>Plan/Preview Usage Guidance</h3>
<ul>
<li>Use these tools after gathering enough context from Read / Observe tools.</li>
<li>Treat plan outputs as review artifacts; apply mutations through explicit Write / Mutate tools only.</li>
<li>In strict MCP policy mode, these tools are denied by design.</li>
</ul>
<h3>Read/Observe Usage Guidance</h3>
<ul>
<li>Use these tools to understand the repository, verify state, and gather evidence.</li>
<li>Never use them to make or apply changes.</li>
<li>If the agent is asked to recommend a fix or document a problem, keep the output descriptive and defer mutation to the Write / Mutate group.</li>
</ul>
<h2>Write / Mutate Tools</h2>
<p>These tools change repository state, notes, or bundle planning metadata. Expose them only when the agent is explicitly authorized to modify content.</p>
<ul>
<li><code>replace_repomix_span</code></li>
<li><code>notes_new</code></li>
<li><code>notes_update</code></li>
<li><code>notes_rename</code></li>
<li><code>notes_delete</code></li>
</ul>
<h3>Write/Mutate Usage Guidance</h3>
<ul>
<li>Require explicit user authorization or a strong policy before enabling these tools.</li>
<li>Use the Read / Observe group first to gather context and confirm the target location.</li>
<li>Avoid requesting multiple mutations in a single turn unless the task is clearly defined.</li>
</ul>
<h2>Example Prompt Pattern</h2>
<p>Use this format when constructing a system prompt for an agent that consumes the MCP tool set:</p>
<pre><code class="language-text">You have access to two tool categories:

1. Read / Observe tools for safe inspection and context gathering.
2. Plan / Preview tools for deterministic planning and bundle metadata.
3. Write / Mutate tools for explicit, authorized changes.

Start by using Read / Observe tools to locate the relevant file, section, or note. Use Plan / Preview tools to confirm deterministic outputs, then request Write / Mutate tools only with high confidence in the exact target.
</code></pre>
<h2>Recommended Policy</h2>
<ul>
<li>Default to Read / Observe mode for all sessions.</li>
<li>Require an explicit permission token or user affirmation before enabling Write / Mutate.</li>
<li>Prefer <code>notes_new</code>, <code>notes_update</code>, and <code>replace_repomix_span</code> only after a successful <code>read</code> or <code>inspect</code> operation.</li>
</ul>
<h2>Why This Doc Exists</h2>
<p>The human-facing <code>docs/modules/ROOT/pages/repository/docs/agent_integration.adoc</code> uses the origin-based three-pillar taxonomy for readability and subsystem teaching. This separate machine-oriented doc exists to translate that human structure into a strict intent-based policy layer that agents can use reliably.</p>
