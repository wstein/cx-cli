<p>This guide explains how documentation is organized, maintained, and marked for authority and stability.</p>
<h2>Source-of-Truth Markers</h2>
<p>Every documentation file includes an HTML comment at the top declaring its source, status, and stability:</p>
<pre><code>&lt;!-- Source: FILENAME.md | Status: CANONICAL | Stability: STABLE --&gt;
</code></pre>
<h3>Status Values</h3>
<ul>
<li><strong>CANONICAL</strong>: Authoritative source of truth. Changes here are binding. Other docs derive from this.</li>
<li><strong>DERIVED</strong>: Auto-generated or synced from another source. Do not edit directly; update the source instead.</li>
<li><strong>EDITORIAL</strong>: Explanatory or tutorial content. May lag the canonical source. Lower priority for consistency.</li>
</ul>
<h3>Stability Values</h3>
<ul>
<li><strong>STABLE</strong>: Semver-protected contract. Breaking changes require major version bump (documented in release notes).</li>
<li><strong>BETA</strong>: Functional but subject to change in minor versions. Operators should pin versions if depending on this doc.</li>
<li><strong>EXPERIMENTAL</strong>: Actively developed. May be removed or substantially rewritten without notice.</li>
</ul>

<h2>Docs Surface Budget</h2>
<p><code>cx</code> intentionally keeps a small front-door docs surface.</p>
<h3>Front-Door Docs</h3>
<p>These are the only documents that should behave like primary entrypoints:</p>
<ul>
<li><code>README.md</code></li>
<li><code>docs/README.md</code></li>
<li><code>docs/SYSTEM_MAP.md</code></li>
<li><code>docs/OPERATING_MODES.md</code></li>
<li><code>docs/MANUAL.md</code></li>
</ul>
<h3>Canonical Core Docs</h3>
<p>These own stable meaning and contracts, but they are not additional onboarding
doors:</p>
<ul>
<li><code>docs/MENTAL_MODEL.md</code></li>
<li><code>docs/SYSTEM_CONTRACTS.md</code></li>
<li><code>docs/GOVERNANCE.md</code></li>
</ul>
<h3>Reference-Only Docs</h3>
<p>Everything else should be written as reference, integration detail, workflow
example, or historical material.</p>
<p>That means:</p>
<ul>
<li><code>ARCHITECTURE.md</code> is an implementation reference for contributors, not a
first-read overview.</li>
<li><code>WORKFLOWS/*</code> are special-case execution examples, not parallel onboarding
paths.</li>
<li><code>MIGRATIONS/*</code> and release-history material are historical references, not
part of the operator front door.</li>
<li>Agent integration and configuration docs should be linked by concern, not
treated as general entrypoints.</li>
</ul>
<h3>Default Rule</h3>
<p>If a new document would introduce another plausible place to start, it should
usually become a section inside an existing front-door doc instead.</p>
<hr>
<h2>Hard Hierarchy Contract</h2>
<p><code>cx</code> intentionally uses a hard documentation hierarchy so semantics have one canonical home.</p>
<h3>Canonical Roles</h3>
<ul>
<li><code>MENTAL_MODEL.md</code> owns canonical semantics.</li>
<li><code>OPERATING_MODES.md</code> maps those semantics to operator choices and progressive onboarding.</li>
<li><code>WORKFLOWS/*</code> shows concrete execution examples across time.</li>
<li><code>AGENT_*</code> documents the integration layer: policy consequences, tool sequences, client setup, and review loops.</li>
</ul>
<h3>Non-Negotiable Rule</h3>
<p>Everything outside <code>MENTAL_MODEL.md</code> must reference canonical semantics instead of redefining them.</p>
<p>That means:</p>
<ul>
<li><code>OPERATING_MODES.md</code> should say which mode to choose, not restate the full semantics of the CX triad.</li>
<li><code>WORKFLOWS/*</code> should show commands, decisions, and outcomes, not redefine Track A vs Track B.</li>
<li><code>AGENT_*</code> docs should explain how agents consume the model through MCP and integration boundaries, not fork the semantics into a second source of truth.</li>
<li>General docs such as <code>README.md</code>, <code>MANUAL.md</code>, and <code>ARCHITECTURE.md</code> should point back to the canonical layer with <code>See: MENTAL_MODEL.md</code> where deeper semantics would otherwise be repeated.</li>
</ul>
<p>Why this protects you: once multiple docs compete to define the same concept, semantic drift becomes invisible until operators make the wrong decision from stale text. The hierarchy keeps one semantic source of truth and pushes every other doc toward mapping, execution, or integration.</p>

<h2>Update Procedures</h2>
<h3>Updating Canonical Docs</h3>
<ol>
<li>Make changes directly in the canonical file</li>
<li>Ensure the change doesn&#39;t break the semver contract</li>
<li>Update the git log message to reference the doc change</li>
<li>If a derived doc depends on this content, update it too</li>
</ol>
<h3>Updating Derived Docs</h3>
<p><strong>Do not edit directly.</strong> Instead:</p>
<ol>
<li>Identify the canonical source</li>
<li>Make the change in the source</li>
<li>Re-run the generation/sync process</li>
</ol>
<h3>Updating Editorial Docs</h3>
<p>Changes are welcome. Update the file, no special approval needed.</p>
<hr>
<h2>Stability Policy</h2>
<h3>STABLE Documents</h3>
<p><strong>What it means:</strong></p>
<ul>
<li>Output schema and option names are locked</li>
<li>Deprecations are announced 2+ releases in advance with migration guidance</li>
<li>Breaking changes require major version bump</li>
</ul>
<p><strong>Example:</strong> If STABILITY.md says tool <code>list</code> is STABLE, we cannot remove the tool or rename its output fields without releasing v1.0.</p>
<h3>BETA Documents</h3>
<p><strong>What it means:</strong></p>
<ul>
<li>Functional but may evolve in minor releases</li>
<li>Field names, structure, or guidance may change</li>
<li>Deprecation notice given 1 release in advance</li>
</ul>
<p><strong>Example:</strong> If AGENT_OPERATING_MODEL.md policy tier system were BETA, operators should not depend on exact policy names across versions.</p>
<h3>EXPERIMENTAL Documents</h3>
<p><strong>What it means:</strong></p>
<ul>
<li>Subject to removal or major rewrite</li>
<li>No advance notice required</li>
<li>Do not build long-term dependencies on content</li>
</ul>

<h2>Why This System?</h2>
<p>Without explicit governance, docs drift: some become outdated, some contradict each other, and operators don&#39;t know what to trust.</p>
<p>By marking every doc:</p>
<ul>
<li>Operators know which docs are safe to depend on (CANONICAL + STABLE)</li>
<li>We can deprecate docs intentionally (moving from STABLE to EXPERIMENTAL)</li>
<li>Derived docs stay in sync with their sources</li>
<li>Tutorial content can be approximate (EDITORIAL) without breaking guarantees</li>
</ul>
<p>This is lightweight (one-line comments) but removes ambiguity.</p>
