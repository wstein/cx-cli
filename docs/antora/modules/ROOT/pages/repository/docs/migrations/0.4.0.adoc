= Migration Notes For 0.4.0
:page-layout: default

[.doc-origin]
Source companion: `docs/MIGRATIONS/0.4.0.md`

++++
<p>Use this note when upgrading to <code>cx</code> <code>0.4.0</code>.</p>
<h2>Release contract</h2>
<p><code>cx</code> now explicitly operates as three primary cooperating surfaces:</p>
<ul>
<li>immutable snapshots with <code>cx bundle</code></li>
<li>live agent protocol with <code>cx mcp</code></li>
<li>durable knowledge with <code>cx notes</code></li>
</ul>
<p>Track B generates hypotheses. Track A generates proofs. Notes are the durable cognition layer between them.</p>
<p>The shipped proof path is kernel-owned. The remaining adapter/oracle seam is
for diagnostics, parity visibility, and reference comparison only.</p>
<h2>What changed in 0.4.0</h2>
<h3>1. Coverage and release assurance</h3>
<ul>
<li>Added a dedicated Vitest coverage lane with HTML and Markdown reporting.</li>
<li>Coverage now participates in release assurance and the public Pages surface.</li>
</ul>
<h3>2. MCP debugging cockpit</h3>
<ul>
<li>Added focused Vitest MCP lanes and UI-oriented debugging guidance.</li>
<li>Added <code>cx mcp catalog --json</code> as the narrow machine-readable MCP contract view.</li>
</ul>
<h3>3. Cognition and trust model</h3>
<ul>
<li>Notes governance, contradiction checks, cognition scoring, and trust propagation are now part of the operating model.</li>
</ul>
<h3>4. Canonical operating model</h3>
<ul>
<li>The docs now explicitly define Track B as hypothesis generation and Track A as proof generation.</li>
</ul>
<h3>5. Adapter JSON contract naming</h3>
<ul>
<li>The dedicated <code>cx adapter</code> command owns adapter metadata for machine-readable
automation.</li>
<li>Top-level JSON payload names are treated as versioned contract surfaces and
must not drift silently.</li>
<li><code>cx adapter capabilities --json</code> now exposes explicit <code>oracleAdapter</code> and
<code>referenceAdapter</code> fields instead of the older <code>repomix</code> naming.</li>
<li>CI now includes a single official-<code>repomix</code> reference-oracle smoke lane.</li>
<li>The older <code>@wsmy/repomix-cx-fork</code> package is no longer a shipped runtime
dependency.</li>
<li>The native kernel is now the production proof path for <code>bundle</code>, <code>validate</code>,
<code>verify</code>, and <code>extract</code>.</li>
<li>Other JSON command surfaces such as <code>cx bundle --json</code>, <code>cx inspect --json</code>,
<code>cx list --json</code>, <code>cx validate --json</code>, and <code>cx verify --json</code> no longer
embed adapter metadata.</li>
<li>The config namespace remains <code>[repomix]</code> in <code>cx.toml</code> for this release.</li>
</ul>
<h2>If you use MCP</h2>
<p>MCP remains an evolving integration surface. Re-check local integration settings, capability expectations, and policy boundaries. For machine-readable inspection, prefer <code>cx mcp catalog --json</code>; for broader MCP diagnostics, use <code>cx doctor mcp --json</code>.</p>
<h2>Operator surface summary</h2>
<p>Use <code>0.4.0</code> with these explicit surfaces in mind:</p>
<ul>
<li>native proof path: <code>bundle</code>, <code>validate</code>, <code>verify</code>, <code>extract</code></li>
<li>live workspace path: <code>mcp</code></li>
<li>durable cognition path: <code>notes</code></li>
<li>adapter/oracle path: diagnostics and parity only</li>
</ul>
<h2>If you rely on coverage</h2>
<p>Use the Vitest coverage lane and HTML artifact as the primary reporting surface:</p>
<pre><code class="language-bash">bun run ci:test:coverage
</code></pre>
<p>Vitest coverage is now the authoritative coverage-reporting lane for release assurance. Bun remains part of the execution and compatibility matrix.</p>
<h2>If you maintain notes</h2>
<p>Run the governance path as part of normal review:</p>
<pre><code class="language-bash">cx notes check
bun run ci:notes:governance
</code></pre>
<p>Notes are still conditional knowledge, not proof-grade artifacts. Governance now surfaces cognition, staleness, contradiction, and trust signals more explicitly.</p>
<h2>If you consume schemas</h2>
<p>Verify current schema references and Pages-hosted <code>$id</code> values:</p>
<ul>
<li><code>https://wstein.github.io/cx-cli/schemas/cx-config-v1.schema.json</code></li>
<li><code>https://wstein.github.io/cx-cli/schemas/cx-config-overlay-v1.schema.json</code></li>
<li><code>https://wstein.github.io/cx-cli/schemas/shared-handover-v2.schema.json</code></li>
<li><code>https://wstein.github.io/cx-cli/schemas/json-section-output-v1.schema.json</code></li>
</ul>
<p>If you depend on manifest schemas, confirm which manifest version your downstream tooling expects before upgrading.</p>
<p>For the closed <code>v0.4</code> native-render contract state, the manifest schema is now <code>v9</code>.
The shared handover filename is recorded in <code>handoverFile</code>, and adapter runtime
metadata has been removed from the manifest entirely. Shared-handover JSON
schema <code>v2</code> is the explicit contract version for
<code>recentRepositoryHistory[].message</code>, which preserves full multiline commit
messages instead of a subject-only field shape. Older manifest schemas still
record <code>repomixVersion</code> or <code>adapterVersion</code> in their own versioned history.
Pre-<code>v1.0.0</code>, older published manifest schemas remain available on Pages under
their versioned filenames. The schema history will be reset deliberately when
<code>cx</code> reaches <code>v1.0.0</code>.</p>
