= Internal API Contract
:page-layout: default

[.doc-origin]
Source companion: `docs/INTERNAL_API_CONTRACT.md`

++++
<p>This document records the internal interfaces that <code>cx</code> treats as stable
architecture seams before any plugin or extension system is introduced.</p>
<p>These interfaces are not public package APIs, but they are internal contract
surfaces. Changes to them should be deliberate, reviewed, and accompanied by
tests and documentation updates.</p>
<h2>RenderEngine</h2>
<p>Defined in <a href="https://github.com/wstein/cx-cli/blob/main/src/render/types.ts">src/render/types.ts</a>.</p>
<p><code>RenderEngine</code> owns proof-path section rendering.</p>
<p>Responsibilities:</p>
<ul>
<li>render section output for <code>xml</code>, <code>markdown</code>, <code>json</code>, and <code>plain</code></li>
<li>return output token counts</li>
<li>return file content hashes</li>
<li>return spans where the style requires them</li>
<li>return structured plans and plan hashes when required</li>
</ul>
<p>Rule:</p>
<ul>
<li>proof-path callers depend on <code>RenderEngine</code>, not on a concrete renderer</li>
</ul>
<h2>StructuredRenderPlan</h2>
<p>Defined in <a href="https://github.com/wstein/cx-cli/blob/main/src/render/types.ts">src/render/types.ts</a>.</p>
<p><code>StructuredRenderPlan</code> is the kernel-owned proof model for rendered section
entries.</p>
<p>Responsibilities:</p>
<ul>
<li>stable entry ordering</li>
<li>stable content identity</li>
<li>stable per-entry token counts</li>
<li>language tagging when available</li>
</ul>
<p>Rule:</p>
<ul>
<li>verification logic depends on the structured plan contract, not on adapter
output shapes</li>
</ul>
<h2>TokenizerProvider</h2>
<p>Defined in <a href="https://github.com/wstein/cx-cli/blob/main/src/shared/tokenizer.ts">src/shared/tokenizer.ts</a>.</p>
<p><code>TokenizerProvider</code> owns deterministic token counting.</p>
<p>Responsibilities:</p>
<ul>
<li>count tokens for in-memory text</li>
<li>count tokens for files on disk</li>
<li>provide a stable seam between callers and tokenizer implementations</li>
</ul>
<p>Current default implementation:</p>
<ul>
<li><a href="https://github.com/wstein/cx-cli/blob/main/src/shared/tokens.ts">src/shared/tokens.ts</a></li>
</ul>
<p>Rule:</p>
<ul>
<li>bundle planning, bundle reporting, and inspect reporting should depend on the
provider interface instead of calling tokenizer implementations directly</li>
</ul>
<h2>ScannerPipeline</h2>
<p>Defined in <a href="https://github.com/wstein/cx-cli/blob/main/src/doctor/scanner.ts">src/doctor/scanner.ts</a>.</p>
<p><code>ScannerPipeline</code> owns bundle-stage security scanning.</p>
<p>Responsibilities:</p>
<ul>
<li>scan source files before proof artifacts are finalized</li>
<li>optionally scan rendered bundle artifacts before finalization</li>
<li>return structured findings with:<ul>
<li>scanner id</li>
<li>profile</li>
<li>stage</li>
<li>severity</li>
<li>proof-blocking status</li>
</ul>
</li>
<li>keep scanner selection explicit through known scanner IDs</li>
<li>enforce explicit fail/warn handling</li>
</ul>
<p>Current default implementation:</p>
<ul>
<li>the reference scanner bridge around the installed oracle scanner</li>
</ul>
<p>Current bounded configuration:</p>
<ul>
<li>explicit scanner IDs</li>
<li>source-stage scanning</li>
<li>optional post-pack artifact scanning</li>
</ul>
<p>Non-goals:</p>
<ul>
<li>plugin discovery</li>
<li>open-ended scanner loading</li>
<li>implicit proof-path scanner activation</li>
</ul>
<p>Rule:</p>
<ul>
<li>scanning behavior belongs to the pipeline contract, not to renderer helpers</li>
</ul>
<h2>Adapter-Oriented Expert Surfaces</h2>
<p>The adapter/oracle seam and the <code>cx adapter</code> command family are expert
diagnostic surfaces.</p>
<p>They exist for:</p>
<ul>
<li>parity diagnostics</li>
<li>oracle capability inspection</li>
<li>migration and reference testing</li>
</ul>
<p>They are not part of ordinary proof-path execution, and production bundle or
verification flows must not depend on them.</p>
<h2>Stability Rules</h2>
<ul>
<li>Add new internal interfaces only when they create a durable ownership
boundary.</li>
<li>Refactors that cross these seams must preserve contract tests and runtime
behavior.</li>
<li>Do not introduce plugin-facing variability into these interfaces until the
plugin system itself is explicitly implemented.</li>
</ul>
