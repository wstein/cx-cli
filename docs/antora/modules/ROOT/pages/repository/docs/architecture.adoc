<p>This is the implementation reference for contributors who need to understand how
the codebase enforces the canonical model.</p>
<p>Read this after <a href="mental_model/">MENTAL_MODEL.md</a>,
<a href="system_contracts/">SYSTEM_CONTRACTS.md</a>, and <a href="system_map/">SYSTEM_MAP.md</a>,
not before them.</p>
<h2>Design Goal</h2>
<p><code>cx</code> exists to make context bundling reproducible enough for automation.</p>
<p>For the documentation map, see <a href="readme/">README.md</a>.</p>
<p>The project now centers a native render kernel because rendering alone is not enough for CI/CD. A pipeline also needs deterministic planning, exact metadata, verification, and explicit recovery semantics. The remaining adapter/oracle seam exists for diagnostics and parity visibility, not as the shipped proof path.</p>
<h2>Canonical Model</h2>
<p>See: <a href="operating_modes/">OPERATING_MODES.md</a>
See: <a href="mental_model/">MENTAL_MODEL.md</a></p>
<p>This document starts where the shared mental model stops. The mental model defines the CX triad, Track A vs Track B, MCP policy tiers, and the artifact lifecycle. Architecture explains how the codebase enforces that model with deterministic planning, manifest truth, verification rules, and extraction guardrails.</p>
<h2>Philosophy</h2>
<p><code>cx</code> is an operational bundler with a kernel-owned proof path.</p>
<p>That distinction explains most of the architecture:</p>
<ul>
<li>exploratory packing tolerates heuristics</li>
<li>operational bundling needs reproducibility</li>
<li>local prompt assembly can accept &quot;close enough&quot;</li>
<li>remote automation needs &quot;provably the same&quot;</li>
</ul>
<p>This is why <code>cx</code> records token accounting in the manifest, writes canonical JSON, and protects emitted artifacts with SHA-256 checksums. Those are not decorative constraints. They are the mechanisms that let a bundle survive time, transport, and automation boundaries.</p>
<h2>Current Implementation Priorities</h2>
<p>The current reliability program avoids broad framework churn and instead
prioritizes boundary cleanup and clearer ownership.</p>
<p>The active modernization targets are:</p>
<ul>
<li>keeping command I/O injectable instead of process-global</li>
<li>keeping workspace context explicit instead of ambient</li>
<li>keeping Vitest as the authoritative shared-suite test runner and coverage lane</li>
<li>keeping Bun limited to explicit runtime compatibility smoke</li>
<li>keeping the adapter seam parity-only and out of the shipped proof path</li>
</ul>
<p>That means the main architectural work is still about deterministic boundaries,
not about swapping frameworks for their own sake.</p>
<h2>System Boundary</h2>
<p><code>cx</code> does not shell out to the adapter/oracle runtime. It uses a narrow boundary and only relies on the small public surface it actually needs for diagnostics and parity work.</p>
<p>Core responsibility split:</p>
<ul>
<li>Render kernel (<code>src/render/</code>): proof-path interfaces, ordering, spans, and
plan hashing</li>
<li>Adapter/oracle seam (<code>src/adapter/</code>): parity-only adapter integration and
runtime capability reporting for expert oracle diagnostics during migration</li>
<li><code>cx</code> planner: decide which files belong where</li>
<li><code>cx</code> manifest layer: describe the bundle in stable JSON</li>
<li><code>cx</code> verification layer: confirm artifacts and source-tree alignment</li>
<li><code>cx</code> extraction layer: recover files according to manifest truth</li>
</ul>
<h2>Structured Render Contract</h2>
<p>The canonical proof-path contract now lives in
<a href="render_kernel_contract/"><code>RENDER_KERNEL_CONTRACT.md</code></a>. This section gives
architectural context for the current implementation.</p>
<p>The stabilized internal seams that sit beneath that contract are recorded in
<a href="internal_api_contract/"><code>INTERNAL_API_CONTRACT.md</code></a>.</p>
<p><code>cx</code> enforces a deterministic structured render contract instead of relying on
heuristic span parsing. The native kernel owns the production proof model; the
adapter/oracle seam is comparison-only.</p>
<p>The structured render contract is represented as:</p>
<pre><code class="language-ts">interface StructuredRenderEntry {
  path: string;
  content: string;
  sha256: string;        // Content hash, not dependent on rendering
  tokenCount: number;
}

interface StructuredRenderPlan {
  entries: StructuredRenderEntry[];
  ordering: string[];    // Canonical lexicographic ordering
}
</code></pre>
<h3>Enforcement</h3>
<ol>
<li><p><strong>Deterministic ordering</strong>: All files are sorted lexicographically. <code>validatePlanOrdering()</code> enforces this invariant.</p>
</li>
<li><p><strong>Content integrity</strong>: Each file&#39;s sha256 is computed during structured extraction, not after rendering. <code>validateEntryHashes()</code> verifies consistency.</p>
</li>
<li><p><strong>Plan hash</strong>: The manifest stores <code>renderPlanHash</code>, computed from the deterministic JSON representation of the plan. This provides cryptographic proof that the render plan is correct.</p>
</li>
<li><p><strong>Verification</strong>: <code>cx verify</code> now checks:</p>
<ul>
<li>Ordering is deterministic (no regressions in file order)</li>
<li>All entry hashes are consistent (content didn&#39;t drift)</li>
<li>Plan hash is reproducible (render contract is sound)</li>
</ul>
</li>
</ol>
<h3>Files</h3>
<ul>
<li><code>src/render/types.ts</code>: kernel-owned proof-path types</li>
<li><code>src/render/engine.ts</code>: render engine interface plus the current native-first
default implementation</li>
<li><code>src/render/native/</code>: native proof-path renderers for XML, Markdown, Plain,
and JSON section outputs plus shared render helpers</li>
<li><code>src/render/structuredPlan.ts</code>: kernel-owned structured plan construction for
native rendering plus adapter-plan extraction for parity/oracle use</li>
<li><code>src/render/ordering.ts</code>: deterministic ordering invariant checks</li>
<li><code>src/shared/tokenizer.ts</code>: <code>TokenizerProvider</code> interface plus the default
tokenizer implementation used by bundle, inspect, and render planning</li>
<li><code>src/doctor/scanner.ts</code>: <code>ScannerPipeline</code> interface plus the reference
scanner bridge used by <code>cx doctor secrets</code></li>
<li><code>src/render/planHash.ts</code>: section and aggregate render-plan hashing</li>
<li><code>src/render/spans.ts</code>: style-aware output span helpers</li>
<li><code>src/adapter/oracleRender.ts</code>: reference-oracle renderer retained for parity
checks and compatibility diagnostics</li>
<li><code>src/manifest/types.ts</code>: Added <code>renderPlanHash</code> field</li>
<li><code>src/manifest/build.ts</code>: Computes aggregate plan hash from sections</li>
<li><code>src/bundle/verify.ts</code>: Validates plan integrity during verification</li>
</ul>
<h2>Module Layer Rules</h2>
<p>The codebase enforces strict import directionality. Violations are bugs, not
style issues.</p>
<pre><code class="language-text">config/  shared/  vcs/           ← foundation (no domain imports)
    ↓
notes/   planning/  manifest/    ← domain modules
    ↓
inspect/  doctor/   adapter/     ← cross-domain orchestration
    ↓
mcp/                             ← transport layer (imports domain only)
    ↓
cli/commands/                    ← presentation layer (thin shells)
</code></pre>
<p><strong>Enforced boundaries:</strong></p>
<ul>
<li><p><code>planning/</code> must not import from <code>notes/</code>. The planner classifies files; note
graph enrichment is an orchestration concern. <code>notes/planner.ts</code> provides
<code>enrichPlanWithLinkedNotes</code> as a post-planning step called by the CLI bundle
and inspect paths, not by the planner itself.</p>
</li>
<li><p><code>mcp/</code> must not import from <code>cli/commands/</code>. MCP is a transport layer; it
imports domain functions from <code>doctor/</code>, <code>inspect/</code>, <code>notes/</code>, and
<code>planning/</code> directly. The CLI command files are thin presentation shells that
re-export from those same domain modules.</p>
</li>
<li><p>Note CRUD operations (<code>createNewNote</code>, <code>readNote</code>, <code>updateNote</code>, <code>renameNote</code>,
<code>deleteNote</code>, <code>searchNotes</code>, <code>listNotes</code>) live in <code>notes/crud.ts</code>, not in
<code>cli/commands/</code>. CLI and MCP both import from the domain module.</p>
</li>
<li><p>MCP config resolution (<code>resolveMcpConfigPath</code>) lives in <code>mcp/config.ts</code>, not
in <code>cli/commands/</code>. The CLI mcp command re-exports it from there.</p>
</li>
</ul>

<h2>Adapter Boundary</h2>
<p>The <code>src/adapter/</code> directory is the explicit boundary between <code>cx</code> core logic
and the external oracle/reference runtime.</p>
<p>This is an expert surface:</p>
<ul>
<li>parity diagnostics</li>
<li>oracle capability inspection</li>
<li>migration and reference testing</li>
</ul>
<p>It is not required for ordinary bundle, verify, validate, or extract flows.</p>
<ul>
<li><code>oracleRender.ts</code> owns parity-only calls into the adapter. Nothing outside
this module invokes adapter render functions directly.</li>
<li><code>capabilities.ts</code> performs runtime feature detection (is <code>packStructured</code>
 available? is <code>renderWithMap</code> available?) so the planner and manifest builder
 never need to know which rendering path was taken.</li>
<li><code>section.ts</code> provides the render-command wrapper for explicit section output
without letting the CLI reach into render internals directly.</li>
</ul>
<p>The <code>[repomix]</code> section in <code>cx.toml</code> is adapter-specific configuration.
<code>cx.toml</code> keys like <code>show_line_numbers</code>, <code>include_empty_directories</code>, and
<code>security_check</code> are passed through exclusively within adapter/scanner bridge
code and are never interpreted by the planner, manifest builder, or any other
core proof-path module. <code>style</code> is the single key shared between the two
layers: it is used by the planner to determine output file extensions and by
adapter/oracle code only when comparison tooling needs it.</p>
<p>Future external comparison backends should follow the same pattern: a new
<code>src/&lt;backend&gt;/</code> directory that exposes the same comparison seam while keeping
core modules unaware of adapter internals.</p>
<h3>Current Migration State</h3>
<p>The proof path now uses a native-only production engine:</p>
<ul>
<li>XML, Markdown, Plain, and JSON sections are rendered through the native
kernel path in <code>src/render/native/</code></li>
<li><code>tests/render/nativeParity.test.ts</code> is the release-gating evidence that the
reference oracle and native kernel still satisfy the same proof contract</li>
</ul>
<p>This split is intentional. It lets the kernel own the production proof path
without weakening extraction, verification, or manifest trust. Official
<code>repomix</code> remains only as an optional reference oracle for parity/testing work;
the older fork is historical.</p>
<h2>Pipeline</h2>
<h3>1. Configuration load</h3>
<p><code>cx</code> loads <code>cx.toml</code>, expands supported paths, resolves behavioral settings, and validates the project configuration.</p>
<h3>2. Deterministic planning</h3>
<p>Planning is a three-phase pipeline anchored to the VCS-derived master list.</p>
<h4>Phase 1: Build the master base</h4>
<p>The planner resolves the source of truth for candidate files from version
control, not from a broad filesystem walk. The priority order is:</p>
<ol>
<li><strong>Git</strong> — <code>git ls-files --cached</code> provides the tracked set.</li>
<li><strong>Fossil</strong> — <code>fossil ls</code> provides the tracked set.</li>
<li><strong>Filesystem</strong> — used as a fallback when no VCS root is detected (for test
 environments or unversioned workspaces).</li>
</ol>
<p>That VCS-derived list is the master base for all later planning decisions.</p>
<blockquote>
<p>Tests validate VCS dispatch using real temporary repositories rather than module mocking.
This avoids worker-global mock leakage and keeps the planner behavior identical to the actual
Git/Fossil/Hg provider code paths used in production.</p>
</blockquote>
<h4>Phase 2: Apply global list shaping</h4>
<p>The global <code>[files] include</code> array can extend the master base with extra paths
that the VCS does not track, such as generated artefacts or build outputs.
The global <code>[files] exclude</code> array is applied after all extensions to remove
paths that should never be planned.</p>
<p>This shaping step changes membership in the master base, but it still does not
let sections discover new files on their own.</p>
<h4>Phase 3: Classify into sections</h4>
<p>Section <code>include</code> and <code>exclude</code> globs are <strong>classifiers</strong>, not discoverers.
They operate only on the already-computed master base and can never add a file
that is not already in it. This separates the question of <em>what exists</em> from
the question of <em>where it belongs</em>.</p>
<p>At this stage the planner resolves:</p>
<ul>
<li>project name</li>
<li>source root</li>
<li>output directory</li>
<li>VCS provider and working-tree dirty state</li>
<li>section membership (including priority-based overlap resolution)</li>
<li>copied assets</li>
<li>unmatched files</li>
<li>overlap and collision failures</li>
</ul>
<p>This happens before rendering because the plan must be settled first.</p>
<h4>Dirty state taxonomy</h4>
<p>After deriving the master list, the planner classifies the working tree into
one of four states:</p>
<table>
<thead>
<tr>
<th>State</th>
<th>Condition</th>
<th>Default behavior</th>
</tr>
</thead>
<tbody><tr>
<td><code>clean</code></td>
<td>No modified or untracked files</td>
<td>Plan proceeds normally</td>
</tr>
<tr>
<td><code>safe_dirty</code></td>
<td>Only untracked files present</td>
<td>Plan proceeds with a warning</td>
</tr>
<tr>
<td><code>unsafe_dirty</code></td>
<td>Tracked files have uncommitted modifications</td>
<td>Planning aborts (exit 7)</td>
</tr>
<tr>
<td><code>forced_dirty</code></td>
<td><code>unsafe_dirty</code> overridden with <code>--force</code></td>
<td>Plan proceeds with a warning</td>
</tr>
<tr>
<td><code>ci_dirty</code></td>
<td><code>unsafe_dirty</code> overridden with <code>--ci</code></td>
<td>Plan proceeds with a warning</td>
</tr>
</tbody></table>
<p>The <code>unsafe_dirty</code> guard exists because a bundle built from a dirty tracked
file cannot be reliably reproduced or verified later. Two escape hatches are
available: <code>--force</code> for local experimentation where a human is present, and
<code>--ci</code> for automated pipelines. Both are recorded in the manifest under
distinct state labels so audit tooling can distinguish human overrides from
pipeline overrides.</p>
<p>Why this protects you: a tracked-file bundle built from a moving working tree cannot later prove what source state it represents. Refusing to proceed by default keeps the artifact contract anchored to reproducible input.</p>
<p>VCS state is not tracked for filesystem-fallback workspaces. Those always
produce <code>dirtyState = &quot;clean&quot;</code> and <code>vcsProvider = &quot;none&quot;</code>.</p>
<h4>Section priority and overlap resolution</h4>
<p>Sections are ordered by their <code>priority</code> value (descending) before overlap
resolution begins. A file claimed by multiple sections is assigned to whichever
section appears first in the resolved order. Sections without an explicit
priority are treated as priority 0 and their relative order follows
<code>dedup.order</code> (config position or lexical) as a stable tie-breaker.</p>
<h4>Catch-all sections</h4>
<p>A section may set <code>catch_all = true</code> instead of providing an <code>include</code> list.
A catch-all section absorbs all files in the master list that were not claimed
by any other section. It runs last in the planning pipeline, after all
normal sections have consumed their files, and may still apply an <code>exclude</code>
list to filter what it absorbs.</p>
<p>At most one catch-all section is permitted per configuration.</p>
<p>This same mechanism lets repository notes live in the normal planning model.
The default <code>docs</code> section now includes <code>notes/**</code>, so durable human intent is
bundled beside documentation and code-facing Markdown instead of being treated
as a separate side channel.</p>
<p>Three overlap handling strategies are available via <code>dedup.mode</code>:</p>
<ul>
<li><code>fail</code> — planning aborts with an actionable message. <code>cx doctor</code> can propose
 static <code>exclude</code> fixes, or you can set higher <code>priority</code> on the owning section
 and switch to <code>first-wins</code> to avoid static TOML mutations entirely.</li>
<li><code>warn</code> — conflicts are reported to stderr and resolution proceeds using
 priority order.</li>
<li><code>first-wins</code> — overlaps are resolved silently using priority order.</li>
</ul>
<p>For rapidly evolving codebases where new files frequently match multiple
sections, <code>dedup.mode = &quot;first-wins&quot;</code> with explicit <code>priority</code> values is
preferable to accumulating static <code>exclude</code> paths in <code>cx.toml</code>. Static excludes
become stale as files are renamed or moved, and they generate merge conflicts
when multiple developers resolve overlaps concurrently on separate branches.</p>
<h3>3. Section rendering</h3>
<p>Each section is rendered as one Repomix-compatible output file in the configured style:</p>
<ul>
<li><code>xml</code></li>
<li><code>markdown</code></li>
<li><code>json</code></li>
<li><code>plain</code></li>
</ul>
<p><code>cx</code> also supplies a section-specific Repomix header through the documented <code>output.headerText</code> option so the file itself carries cx-oriented handover context without post-processing the generated output.
That header is intentionally a presentation-layer prolog rather than part of the core format semantics: it explains what the artifact is, how to interpret logical paths, where edits belong, and which bundle contracts remain authoritative.</p>
<p>This yields more semantic fluency for humans and AI consumers without relaxing the deterministic contract:</p>
<ul>
<li>rigid core semantics for markers, directives, manifests, and validation</li>
<li>expressive consumer guidance in the generated prolog and shared handover</li>
<li>optional narrative cues that never override canonical bundle data</li>
</ul>
<p>The renderer also reports output token counts. If the adapter supports exact span capture, <code>cx</code> records absolute <code>outputStartLine</code> and <code>outputEndLine</code> values for each packed text file in XML, Markdown, and plain sections. Those spans are the primary lookup path for those text formats. JSON uses direct object lookup instead of span metadata, and JSON-only bundles may omit spans entirely.</p>
<h3>4. Shared handover</h3>
<p><code>cx bundle</code> writes a kernel-rendered shared handover file alongside the section
outputs. The handover is meant to travel with the section files when multiple
outputs are handed over together, so the shared context is externalized without
breaking the self-contained section files.</p>
<p>For XML-style bundles, that handover intentionally stays mostly plain text.
Rare tags such as <code>&lt;section_inventory&gt;</code> and <code>&lt;recent_repository_history&gt;</code> act
as semantic anchors for LLMs and operators rather than as a full XML document
contract.</p>
<h3>5. Manifest build</h3>
<p><code>cx</code> writes a canonical manifest that records:</p>
<ul>
<li>bundle identity and versions</li>
<li>source root and bundle directory</li>
<li>VCS provider (<code>git</code>, <code>fossil</code>, or <code>none</code>)</li>
<li>dirty state at bundle time (<code>clean</code>, <code>safe_dirty</code>, or <code>forced_dirty</code>)</li>
<li>list of uncommitted modified files when <code>forced_dirty</code></li>
<li>checksum algorithm</li>
<li>the shared handover filename (currently stored in the manifest <code>handoverFile</code> field)</li>
<li>section outputs</li>
<li>copied assets</li>
<li>repository note metadata, including extracted summaries when notes are present</li>
<li>per-file token counts</li>
<li>source metadata such as size, media type, and mtime</li>
<li>output spans for XML, Markdown, and plain sections when exact span capture is available; JSON sections do not need them</li>
</ul>
<p>The manifest is not just a report. It is the contract other commands operate against.</p>
<h3>6. Lock file and checksums</h3>
<p><code>cx bundle</code> also writes:</p>
<ul>
<li>a lock file containing the resolved Category B behavioral settings used during bundle creation</li>
<li>a SHA-256 checksum sidecar covering the manifest, lock file, section outputs, and copied assets</li>
</ul>
<h3>7. Post-build consumers</h3>
<p>After bundling, other commands use the recorded state:</p>
<ul>
<li><code>validate</code> checks bundle structure and schema</li>
<li><code>verify</code> checks integrity and optional source-tree drift</li>
<li><code>list</code> surfaces stored metadata</li>
<li><code>extract</code> reconstructs files according to manifest semantics</li>
</ul>
<h2>Invariants</h2>
<p>Some failures are fundamental and intentionally hard:</p>
<ul>
<li>section overlap when overlap failure mode is active</li>
<li>asset collision between copied assets and packed files</li>
<li>missing core adapter contract</li>
<li><code>unsafe_dirty</code> working tree without <code>--force</code> (exit code 7)</li>
</ul>
<p>These are Category A invariants. They are never configurable away because doing so would make the bundle ambiguous or unverifiable.</p>
<h2>Category B Behaviors</h2>
<p>Some operational friction points are configurable:</p>
<ul>
<li>overlap handling mode</li>
<li>missing cx-specific adapter extension mode</li>
<li>duplicate config entry mode</li>
</ul>
<p>These are recorded in the lock file so later verification can detect drift between the settings used to build the bundle and the settings currently in effect.</p>
<h2>Why Persistent Token Accounting Matters</h2>
<p>Repomix can already calculate token counts while rendering. <code>cx</code> adds a different guarantee: those counts are carried forward as part of the artifact contract instead of disappearing with a single run.</p>
<p>That matters in automation because later verification and downstream tooling can read the manifest&#39;s recorded token counts directly instead of re-running a render or relying on a fresh estimate in a different environment.</p>
<h2>Why SHA-256 Matters</h2>
<p>Checksums are not included for cryptography theater. They prove that the artifacts the runner sees are the same artifacts the bundler emitted.</p>
<p>Why this protects you: checksum failures are evidence that the artifact set in hand is no longer provably the one <code>cx</code> wrote. Verification stops before a tampered, partial, or substituted bundle can pass as authoritative.</p>
<p>That lets <code>verify</code> detect:</p>
<ul>
<li>manifest tampering</li>
<li>section output drift</li>
<li>missing checksum entries</li>
<li>copied asset mutation</li>
</ul>
<p>For packed text rows, the manifest hash covers the normalized packed content emitted by Repomix. That keeps verification aligned with the actual handover payload instead of pretending <code>cx</code> is a source-byte archiver.</p>
<h2>Why Output Spans Matter</h2>
<p>When exact span capture is available, the manifest can tell downstream tooling where each file lives in the rendered section output.</p>
<p>Those spans are only useful if the output remains deterministic. That is why degraded extraction is treated carefully: once the parser can no longer reconstruct the packed output cleanly, absolute coordinates can become unsafe for downstream automation.</p>
<h2>Config Safety and Merge Semantics</h2>
<p>Configuration files often inherit from other configurations (e.g., project-specific settings extending organization defaults). <code>cx</code> enforces explicit merge semantics to prevent silent overwrites and make configuration conflicts visible.</p>
<h3>Merge Rules</h3>
<p>When merging two configurations (base and override), <code>cx</code> applies these rules:</p>
<ul>
<li><strong>Scalars</strong>: Override wins (right overwrites left). Conflicting scalar values are recorded as conflicts.</li>
<li><strong>Arrays</strong>: Append-only semantics (never silent replace). When both base and override have non-empty arrays, they are concatenated. This prevents accidentally dropping existing patterns or values.</li>
<li><strong>Objects</strong>: Deep merge (recursive application of rules). Nested structures are merged field-by-field.</li>
<li><strong>Undefined</strong>: Treated as &quot;not set&quot;. Missing fields in override do not affect base values.</li>
<li><strong>Null</strong>: Valid overwrite value. Explicit null in override overwrites base (unlike undefined).</li>
</ul>
<h3>Conflict Detection</h3>
<p>Every merge operation returns a conflict list documenting:</p>
<ul>
<li><code>path</code>: The configuration path (e.g., <code>files.exclude</code>, <code>dedup.mode</code>)</li>
<li><code>reason</code>: Why the conflict occurred (e.g., &quot;scalar value replaced&quot;, &quot;array append behavior&quot;)</li>
<li><code>baseValue</code> and <code>overrideValue</code>: The actual conflicting values</li>
</ul>
<p>This explicit logging lets operators audit configuration inheritance chains and detect unintended changes that silent merges would hide.</p>
<h3>Why This Matters</h3>
<p>Configuration is not arbitrary application state. Changes to section definitions, file patterns, or dedup rules affect the reproducibility of a bundle. By making conflicts visible rather than silent, <code>cx</code> ensures that operational decisions about configuration are explicit and auditable.</p>
<h2>Extraction Semantics</h2>
<p><code>cx</code> classifies extraction outcomes as:</p>
<ul>
<li><code>intact</code>: reconstructed text matches the packed-content hash in the manifest</li>
<li><code>copied</code>: asset restored directly from stored bundle content</li>
<li><code>degraded</code>: text is recoverable but does not match the packed-content hash in the manifest</li>
<li><code>blocked</code>: deterministic recovery is not possible from the stored output</li>
</ul>
<p><code>degraded</code> is intentionally not the default success path. It requires explicit operator consent with <code>--allow-degraded</code>.</p>
<p>See <a href="extraction_safety/">Extraction Safety</a> for the operational consequences.</p>
