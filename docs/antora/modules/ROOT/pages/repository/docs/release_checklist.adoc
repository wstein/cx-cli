= Release Checklist
:page-layout: default

[.doc-origin]
Source companion: `docs/RELEASE_CHECKLIST.md`

++++
<p>Use this short checklist when cutting a release.</p>
<ul>
<li>Two-phase release protocol:<ol>
<li>Prepare the release candidate on <code>develop</code>.</li>
<li>Finalize the release with a <code>vX.Y.Z</code> tag from the exact green candidate commit.</li>
</ol>
</li>
<li>Bumping <code>package.json</code> on <code>develop</code> starts the release candidate. Use <code>chore(release): start vX.Y.Z</code> for that commit and treat later commits on the candidate path as release-fix follow-ups only.</li>
<li>Keep fixing forward on <code>develop</code> until the required CI lanes are green. Do not tag early and do not publish from an ordinary <code>develop</code> success alone.</li>
<li>Create and push the release tag from the exact certified <code>develop</code> commit. The release workflow is the finalization step; the version bump is not.</li>
<li>The release workflow must verify that the tag matches <code>package.json</code>, that the tagged commit already passed the required CI gates, and that the published artifacts come from that same certified commit.</li>
<li>Promote <code>main</code> only after successful publishing, and do it with a fast-forward update to the released commit. Do not rebase <code>main</code> onto the release tag.</li>
<li>The release workflow should own that final <code>main</code> fast-forward so the shipped commit and the branch tip stay identical without a manual replay step.</li>
<li>Run <code>make certify</code> before tagging. This runs lint, typecheck, build, full test coverage, the contract lane, the official Repomix reference-oracle smoke lane, bundle transition matrix smoke, release integrity smoke, and a clean double-build reproducibility check. All steps must be green. Use <code>make verify</code> during development; <code>make certify</code> is the pre-tag CI-equivalent gate.</li>
<li>The CI workflow validates Bun against the declared minimum runtime (<code>1.3.11</code>) and <code>latest</code> before release automation is allowed to proceed.</li>
<li>The Pages publish now assembles one site tree with <code>/schemas/</code> and <code>/coverage/</code>; successful <code>main</code> CI publishes the latest coverage page automatically, while tagged releases still mirror schema files into release assets.</li>
<li>Run <code>bun run pages:smoke</code> against the staged <code>dist/site/</code> tree whenever you need to validate the public publish surface locally.</li>
<li>If schema files change, update the published Pages site and verify the <code>schemas/</code> index still lists the current versions.</li>
<li>The publish workflow runs the same smoke check in non-blocking mode before pushing <code>gh-pages</code>, and the separate <code>Pages Smoke</code> workflow keeps that validation visible on pull requests and <code>main</code>.</li>
<li>If the Pages branch does not exist yet, let the publish workflow create <code>gh-pages</code> on the next run.</li>
<li>GitHub Pages must be configured in repository settings to serve from the <code>gh-pages</code> branch.</li>
<li>If a new schema version is added, refresh the public <code>$id</code> values in the JSON Schema files and keep the docs pointing at the GitHub Pages host.</li>
<li>Mirror the same schema files into the GitHub Release assets for the tagged release.</li>
<li>Tagged releases also publish the npm tarball, <code>release-integrity.json</code>, and the generated <code>cx-cli.rb</code> formula into GitHub release assets so the public release payload matches the documented integrity story.</li>
<li>Use <code>make release VERSION=vX.Y.Z</code> as the local release wizard. The first call with a new version starts the candidate on <code>develop</code>; the second call with that same tagged version finalizes the release from the certified commit.</li>
<li>Ensure <code>package.json</code> version matches the git tag before publishing release artifacts.</li>
<li>The release workflow runs on <code>vX.Y.Z</code> tag pushes, plus a manual re-finalization path for an existing tag when needed.</li>
<li>The release workflow only finalizes when the tagged commit already has a successful <code>develop</code> CI run; tag presence without certified CI is not enough.</li>
<li><code>v0.4</code> is now a closed release line. Further work should tighten contracts, docs, or scanner scope on top of that line rather than reopening the architecture transition itself.</li>
<li>The release workflow in <code>.github/workflows/release.yml</code> requires <code>NPM_TOKEN</code> in the <code>node</code> environment so <code>npm publish</code> can authenticate to the npm registry, and <code>HOMEBREW_TAP_PUSH_TOKEN</code> in the <code>homebrew</code> environment so it can authenticate the cross-repo push to <code>wstein/homebrew-tap</code>.</li>
<li>Confirm both environment secrets are set before the release workflow starts; the workflow now fails fast if either one is missing.</li>
<li>Create <code>HOMEBREW_TAP_PUSH_TOKEN</code> as a fine-grained personal access token in GitHub Settings -&gt; Developer settings -&gt; Personal access tokens -&gt; Fine-grained tokens -&gt; Generate new token.</li>
<li>Scope that token to the <code>wstein/homebrew-tap</code> repository only, and grant <code>Contents: Read and write</code> so the release workflow can commit and push <code>Formula/cx-cli.rb</code>.</li>
<li>Store the token in the <code>homebrew</code> environment of the source repo under the name <code>HOMEBREW_TAP_PUSH_TOKEN</code>.</li>
<li>Release order:<ol>
<li>Build the package once.</li>
<li>Publish the packed tarball to npm.</li>
<li>Generate <code>Formula/cx-cli.rb</code> from the same tarball.</li>
<li>Commit the formula into <code>wstein/homebrew-tap</code>.</li>
</ol>
</li>
<li>Final release promotion order:<ol>
<li><code>develop</code> prepares the candidate.</li>
<li>The <code>vX.Y.Z</code> tag finalizes the certified candidate.</li>
<li>The release workflow verifies integrity and publishes from that tagged commit.</li>
<li><code>main</code> fast-forwards to mirror the shipped commit.</li>
</ol>
</li>
<li>Verify the external tap repo <code>wstein/homebrew-tap</code> accepts direct formula updates from the single release workflow, and remember that the tap repo owns formula validation on push.</li>
</ul>
