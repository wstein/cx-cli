= Release Integrity
:page-layout: default

[.doc-origin]
Source companion: `docs/RELEASE_INTEGRITY.md`

++++
<p>cx-cli releases include cryptographic integrity verification to detect unintended changes and ensure builds are reproducible.</p>
<h2>What&#39;s Verified</h2>
<p>Each release publishes a <strong>Software Bill of Materials (SBOM)</strong> with:</p>
<ul>
<li><strong>Release version</strong> — semantic version from package.json</li>
<li><strong>Timestamp</strong> — UTC ISO-8601 timestamp of build</li>
<li><strong>Node version</strong> — Node.js version used to build</li>
<li><strong>Bun version</strong> — Bun runtime version used to build</li>
<li><strong>Tarball name</strong> — published npm tarball filename</li>
<li><strong>Tarball hash</strong> — SHA-256 digest of the tarball</li>
</ul>
<p>The SBOM is generated at build time, staged as <code>tarball-artifacts/release-integrity.json</code>, and published into the GitHub release assets as <code>release-integrity.json</code>.</p>
<h2>Verification Steps</h2>
<h3>1. Obtain the SBOM</h3>
<p>Download <code>release-integrity.json</code> from the GitHub release assets page.</p>
<h3>2. Run Local Verification</h3>
<pre><code class="language-bash">bun run verify-release
</code></pre>
<p>This command:</p>
<ul>
<li>Reads the SBOM from <code>dist/release-integrity.json</code></li>
<li>Locates the tarball in <code>tarball-artifacts/</code></li>
<li>Computes its SHA-256 hash</li>
<li>Compares against the SBOM hash</li>
<li>Reports success or failure with details</li>
</ul>
<h3>3. Manual Verification</h3>
<pre><code class="language-bash"># Compute tarball hash
sha256sum cx-cli-X.Y.Z.tgz

# Compare against SBOM
cat dist/release-integrity.json | jq &#39;.tarballHash&#39;
</code></pre>
<h2>Design</h2>
<h3>Reproducibility</h3>
<p>The integrity system depends on <strong>reproducible builds</strong>: the same source, dependencies, and compiler versions produce byte-identical output.</p>
<p>The CI/CD pipeline includes a <code>reproducibility</code> job that:</p>
<ul>
<li>Builds dist/ twice in sequence</li>
<li>Captures SHA-256 of all dist/*.js files</li>
<li>Fails if any file differs between runs</li>
</ul>
<p>This catches timestamp injection, non-deterministic compilation, or other drift.</p>
<h3>Trust Model</h3>
<p>The SBOM is released alongside the npm tarball and generated Homebrew formula as GitHub release assets. Operators can:</p>
<ol>
<li>Verify the tarball&#39;s SHA-256 matches the SBOM</li>
<li>Confirm build environment (Node, Bun versions)</li>
<li>Audit the release timestamp</li>
</ol>
<p>The SBOM itself is <strong>not signed</strong>. For full cryptographic verification, operators should validate the GitHub release against the repository&#39;s public keys or use GitHub&#39;s own release verification.</p>
<h2>Interpreting Failures</h2>
<p><strong>Hash mismatch</strong> → Tarball has been modified or corrupted. Do not use; re-download from the official GitHub release.</p>
<p><strong>Tarball not found</strong> → Missing or renamed tarball. Verify you have the correct artifact.</p>
<p><strong>SBOM missing or corrupt</strong> → Release metadata is missing. Contact the maintainers.</p>
<h2>Release Integrity Artifact Retention</h2>
<p>The SBOM is published as a GitHub release asset alongside the npm tarball and Homebrew formula. Schema files are mirrored into the same release through the tagged Pages/schema workflow. GitHub retains these assets for the lifetime of the release.</p>
<h2>Future Enhancements</h2>
<p>Planned improvements:</p>
<ul>
<li>GPG signing of release tarballs</li>
<li>Signed attestation of reproducible builds</li>
<li>Supply chain integrity proofs (SLSA provenance)</li>
</ul>
