cx init always generates the shared workspace bootstrap files plus a language-aware Makefile. Some templates also specialize the default authoring-oriented cx-mcp.toml overlay, and TypeScript currently adds a second build-artifact overlay.

The universal generated files are:

  • README.md

  • cx.toml

  • .editorconfig

  • Makefile

  • cx-mcp.toml

  • .mcp.json

  • .vscode/mcp.json

  • .claude/settings.json

  • .codex/settings.json

  • antora-playbook.yml

  • docs/ Antora scaffold with onboarding, manual, and arc42 architecture surfaces

  • notes/ scaffolding

The generated docs scaffold is intentionally minimal:

  • onboarding gives the front door for new contributors and operators

  • manual keeps task-oriented procedures separate from design material

  • architecture uses an arc42 chapter spine so teams can grow the architecture docs incrementally without inventing their own structure first

Capability Levels

  • basic: workspace-aware Makefile plus the shared MCP overlay contract.

  • enhanced: specialized authoring overlay and richer local quality targets.

  • advanced: enhanced template plus separate MCP overlays for authoring and build-artifact inspection.

Overlay Intent

  • Authoring overlay: live source exploration and editing in MCP sessions.

  • Build overlay: compiled or emitted artifacts that should stay separate from the default authoring surface.

If a repository uses nonstandard source or output layouts, edit the overlays independently instead of broadening the default authoring view.

Enhanced Templates

Rust

Use the Rust template for Cargo workspaces that need a richer local verification gate than the basic templates.

Generated targets:

  • make build → cargo build --release

  • make test → cargo test

  • make check → cargo check

  • make verify → cargo fmt --check, cargo clippy --all-targets — -D warnings, cargo test, cargo build --release

  • make certify → verify

Default MCP view:

  • includes src/, tests/, benches/**, Cargo.toml, Cargo.lock, and README.md

  • excludes target/** so build artifacts do not leak into the default authoring view

Go

Use the Go template for module-aware repositories that already center on go.mod.

Generated targets:

  • make build → go build ./…​

  • make test → go test ./…​

  • make check → go vet ./…​

  • make verify → check + test + build

  • make certify → verify

Default MCP view:

  • includes cmd/, internal/, pkg/**, go.mod, go.sum, and README.md

  • should be widened deliberately if your project keeps important source outside those conventional directories

Elixir

Use the Elixir template for Mix-based applications that benefit from a source-first MCP overlay.

Generated targets:

  • make build → mix compile

  • make test → mix test

  • make check → mix format --check-formatted

  • make verify → check + test + build

  • make certify → verify

Default MCP view:

  • includes lib/, test/, mix.exs, mix.lock, and README.md

  • excludes _build/, deps/, and coverage output so MCP stays focused on authoring files by default

Crystal

Use the Crystal template for shard.yml workspaces that want a small, source-oriented MCP surface.

Generated targets:

  • make build → shards build when shard metadata exists, otherwise crystal build src/main.cr --release

  • make test → crystal spec

  • make verify → test + build

  • make certify → verify

Default MCP view:

  • includes src/, spec/, shard.yml, shard.lock, and README.md

  • excludes dependency and build caches such as .shards/, lib/, and bin/**

Zig

Use the Zig template for repositories centered on build.zig.

Generated targets:

  • make build → zig build

  • make test → zig build test

  • make verify → test + build

  • make certify → verify

Default MCP view:

  • includes src/**, build.zig, build.zig.zon, and README.md

  • excludes zig-cache/, .zig-cache/, and zig-out/** to keep the default MCP surface source-first

Advanced Template

TypeScript

Use the TypeScript template when you need the most operationally rich template profile, including package-manager detection and separate MCP overlays for authoring and build artifacts.

Package-manager detection:

  1. bun.lockb or bun.lock

  2. pnpm-lock.yaml

  3. yarn.lock

  4. package-lock.json or npm-shrinkwrap.json

  5. npm when no lockfile exists

This keeps the generated Makefile lockfile-first instead of choosing Bun only because it happens to be installed on the local machine.

Generated targets:

  • make install

  • make build

  • make test

  • make check

  • make lint

  • make verify

  • make certify

TypeScript keeps install separate from build. Build commands should not implicitly mutate dependency state. make lint and make check skip with an explicit message when the corresponding package scripts are not present. make certify runs the package-manager certify script when one exists and otherwise falls back to make verify.

Default MCP view:

  • cx-mcp.toml stays source-oriented for src/**, workspace metadata, and key TypeScript config files

  • cx-mcp-build.toml stays build-oriented for dist/** plus minimal package metadata

Use the source-oriented overlay for live editing and code understanding. Use the build overlay when you want to inspect emitted artifacts without broadening the default authoring surface.

Customize the generated overlays if your repository:

  • uses a source root other than src/

  • emits artifacts outside dist/

  • stores important workspace metadata in additional files

  • needs stricter excludes for framework-specific caches
